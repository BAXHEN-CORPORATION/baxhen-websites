# Baxhen Platform — First-Stage Architecture Implementation Brief

You are working as the lead software architect and senior full-stack engineer for the initial Baxhen website platform.

Your task is to create the first-stage architecture and everything technically necessary before onboarding the first paying client.

Do not build a speculative enterprise platform. Build a clean, secure, tested foundation that can support Baxhen’s first five clients and can be extended later without requiring a complete rewrite.

Before changing files:

1. Inspect the existing repository.
2. Read the current `package.json`, Payload configuration, Next.js configuration, TypeScript configuration, environment files, collections, routes, and tests.
3. Report any conflict between this brief and the existing code.
4. Produce a concise implementation plan.
5. Then implement the plan in small, verifiable stages.
6. Run type checking, linting, integration tests, and the production build after implementation.
7. Do not silently remove existing functionality.

---

# 1. Business Context

Baxhen creates websites for small and medium businesses.

The first commercial offer may include website implementation at no additional cost when the client purchases the annual Baxhen infrastructure plan.

The system must initially support two website types:

## Business Presence Website

A multi-page website intended to:

* Explain the business.
* Present services, products, menus, rooms, portfolios, or other content.
* Establish credibility.
* Provide contact information.
* Support Portuguese and English content.
* Support basic technical SEO.
* Generate calls, messages, or inquiries.

## Lead-Generation Website

A focused website or landing page intended to:

* Promote one service, offer, or campaign.
* Receive traffic from advertisements or social media.
* Present a specific value proposition.
* Collect qualified leads.
* Track form submissions and conversion events.
* Support a shorter and more focused user journey.

Both website types must use the same technical platform and reusable section system. Do not create two separate frameworks.

# 3. Architectural Pattern

Use the following separation:

```text
Model
  ↓
Repository
  ↓
Service
  ↓
ViewModel / Presenter
  ↓
View
```

Next.js route files, layouts, Server Components, route handlers, and Server Actions act as the composition or controller layer.

Do not implement traditional class-based MVC.

## 3.1 Model Layer

The Model layer contains:

* Payload collection definitions.
* Payload global definitions.
* Generated Payload types.
* Domain types.
* Domain enums.
* Validation rules.
* Relationships.
* Collection hooks that protect data integrity.
* Access-control rules.

The Model layer must not import React components.

Examples:

```text
Tenants
Users
Clients
Sites
Pages
Media
Forms
FormSubmissions
Domains
ServicePlans
Contracts
ContractDocuments
Deployments
```

## 3.2 Repository Layer

The Repository layer is responsible only for data access.

Examples:

```ts
findSiteByHostname(hostname)
findSiteBySlug(siteSlug)
findPageByPath(siteId, locale, path)
findNavigation(siteId, locale)
findSiteSettings(siteId, locale)
createFormSubmission(input)
findClientContracts(clientId)
```

Repository rules:

* Payload Local API calls belong here.
* Every public website query must be scoped to a specific site or tenant.
* Never perform an unrestricted query and filter the records afterward in application memory.
* Never trust a tenant or site ID supplied by the browser without validation.
* Avoid returning complete Payload documents when the consumer needs only a small subset.
* Keep database details out of React components.
* Use generated Payload types.
* Handle “not found” conditions explicitly.
* Do not leak draft, private, billing, contract, or cross-tenant information.

Suggested path:

```text
src/repositories/
```

## 3.3 Service Layer

The Service layer contains application and business logic.

Examples:

```ts
resolveWebsiteRequest(hostname, pathname, locale)
buildWebsitePage(site, page, navigation)
submitLeadForm(input, requestMetadata)
determineEnabledLocales(site)
createCanonicalURL(site, locale, path)
createWebsiteSEO(site, page, locale)
provisionDemoTenant()
```

Service rules:

* Services may call repositories.
* Services must not render React.
* Services must not depend on a particular page component.
* Services should be independently testable.
* Services should return explicit success or failure results where failure is expected.
* Business rules must not be duplicated across route files or components.

Suggested path:

```text
src/services/
```

## 3.4 ViewModel and Presenter Layer

The ViewModel layer transforms domain data into presentation-ready data.

Examples:

```ts
toHeaderViewModel(site, navigation, locale)
toFooterViewModel(site, settings, locale)
toHeroViewModel(block, context)
toServiceGridViewModel(block, context)
toContactFormViewModel(form, context)
toSEOViewModel(site, page, locale)
toPageViewModel(site, page, locale)
```

ViewModels should:

* Resolve localized values.
* Apply safe defaults.
* Normalize image data.
* Normalize links.
* Remove CMS-specific field shapes from the View.
* Expose only the values the View requires.
* Create stable component props.
* Avoid returning Payload request objects or Local API objects.
* Avoid querying the database.

Suggested paths:

```text
src/view-models/
src/presenters/
```

Use one of these names consistently. Prefer `view-models`.

## 3.5 View Layer

The View layer contains React components.

Views must:

* Receive data through typed props.
* Remain unaware of Payload.
* Not call the Payload Local API.
* Not query Supabase.
* Not determine tenant permissions.
* Not contain billing or contract logic.
* Not access secrets.
* Be reusable across business-presence and lead-generation sites.
* Use Server Components by default.
* Use Client Components only when browser interaction or browser APIs are genuinely required.

The View layer should be divided into:

```text
Primitives
Components
Sections
Templates
Pages
```

### Primitives

Small visual elements:

```text
Button
Link
Container
Heading
Text
Image
Icon
Stack
Grid
Badge
```

### Components

Reusable interface units:

```text
Logo
LanguageSwitcher
ContactDetails
SocialLinks
FormField
NavigationMenu
MobileNavigation
BusinessHours
```

### Sections

Reusable page-level content sections:

```text
HeroSection
ContentSection
ServicesSection
GallerySection
TestimonialsSection
FAQSection
CallToActionSection
ContactSection
LeadFormSection
MapSection
LogoCloudSection
TeamSection
PricingSection
```

### Templates

Compositions for business use cases:

```text
BusinessPresenceTemplate
LeadGenerationTemplate
LegalPageTemplate
ContactPageTemplate
```

A template should compose sections. It must not fetch CMS data directly.

# 7. Hostname and Site Resolution

Use `src/proxy.ts` because the repository uses a `src` directory.

The proxy should:

1. Read the request hostname.
2. Normalize the hostname.
3. Allow Baxhen’s own application routes to continue normally.
4. Rewrite client website requests to an internal site route.
5. Exclude Payload Admin, API routes, Next.js assets, static files, and health checks.
6. Avoid database calls directly inside Proxy.
7. Pass the normalized hostname to the internal route through a rewrite-safe method.
8. Handle unknown domains with a site-not-found experience.

Do not hardcode each client inside `proxy.ts`.

Because Proxy should not perform the Payload database lookup, use this flow:

```text
Incoming hostname
    ↓
Proxy rewrites to internal website route
    ↓
Website resolution service receives normalized hostname
    ↓
Domain repository resolves active Domain and Site
    ↓
Page service resolves locale and path
    ↓
ViewModel transforms CMS data
    ↓
Template renders reusable sections
```

Support these local-development formats:

```text
demo.localhost:3000
lead-demo.localhost:3000
```

Also support an environment-variable fallback when subdomain-based localhost behavior is unavailable.


Use:

* Payload localization for editable website content.
* `next-intl` for frontend interface strings and locale routing.
* Payload translations for Admin Panel localization.

Requirements:

* Every site has `defaultLocale`.
* Every site has `enabledLocales`.
* Unsupported locales return a valid fallback or not-found response according to a single documented rule.
* Language switching preserves the current page when a translation exists.
* Localized SEO metadata is supported.
* Canonical URLs are locale-aware.
* Alternate-language URLs can be generated.
* Forms display localized labels and messages.
* Do not duplicate an entire page record solely to provide another language unless there is a clear reason.

Create a localization service so locale fallback logic is not duplicated across components.

# 9. Reusable Block and Section System

Payload blocks define editable content structure.

React sections render the visual result.

Keep these concepts separate.

Example:

```text
Payload block configuration:
src/blocks/Hero/config.ts

Payload block type:
HeroBlock

ViewModel:
src/view-models/blocks/hero.view-model.ts

View:
src/views/sections/HeroSection.tsx
```

Create an explicit typed block renderer:

```ts
type SectionViewModel =
  | HeroSectionViewModel
  | ContentSectionViewModel
  | ServicesSectionViewModel
  | GallerySectionViewModel
  | TestimonialsSectionViewModel
  | FAQSectionViewModel
  | CallToActionSectionViewModel
  | ContactSectionViewModel
  | LeadFormSectionViewModel
```

Avoid a block renderer containing large amounts of transformation logic.

The renderer should only map a view-model discriminator to a section component.

Each section must support:

* Consistent spacing.
* A stable container system.
* Responsive layouts.
* Optional theme variants.
* Accessible heading levels.
* Typed CTA links.
* Localized content.
* Sensible empty-state behavior.
* No dependency on Payload request objects.

Build these initial sections:

1. Hero.
2. Rich content.
3. Services or features.
4. Gallery.
5. Testimonials.
6. FAQ.
7. Call to action.
8. Contact information.
9. Contact form.
10. Lead-generation form.

Do not build a drag-and-drop visual page editor.

# 27. Code Quality Rules

* Prefer small pure functions.
* Prefer composition over inheritance.
* Avoid giant collection configuration files.
* Avoid giant React components.
* Avoid business logic inside JSX.
* Avoid CMS queries inside sections.
* Avoid `any`.
* Avoid non-null assertions unless proven safe.
* Avoid duplicated locale fallback logic.
* Avoid duplicated tenant filters.
* Avoid passing complete Payload documents through the entire UI.
* Avoid exposing private fields in public view models.
* Use explicit names rather than generic `utils.ts` files.
* Add comments only where the reason is not evident from the implementation.
* Keep functions focused on one responsibility.
* Keep public interfaces small.
* Ensure errors are actionable.
* Preserve exact Payload package-version alignment.
